Стек: HTML, SCSS, TS, Vite

Структура проекта:
- src/ — исходные файлы проекта
- src/components/ — папка с JS компонентами
- src/components/base/ — папка с базовым кодом

- src/utils/utils.ts — файл с утилитами

# Интернет-магазин «Web-Larёk»
«Web-Larёk» — это интернет-магазин с товарами для веб-разработчиков, где пользователи могут просматривать товары, добавлять их в корзину и оформлять заказы. Сайт предоставляет удобный интерфейс с модальными окнами для просмотра деталей товаров, управления корзиной и выбора способа оплаты, обеспечивая полный цикл покупки с отправкой заказов на сервер.

## Архитектура приложения

Код приложения разделен на слои согласно парадигме MVP (Model-View-Presenter), которая обеспечивает четкое разделение ответственности между классами слоев Model и View. Каждый слой несет свой смысл и ответственность:

Model - слой данных, отвечает за хранение и изменение данных.  
View - слой представления, отвечает за отображение данных на странице.  
Presenter - презентер содержит основную логику приложения и  отвечает за связь представления и данных.

Взаимодействие между классами обеспечивается использованием событийно-ориентированного подхода. Модели и Представления генерируют события при изменении данных или взаимодействии пользователя с приложением, а Презентер обрабатывает эти события используя методы как Моделей, так и Представлений.

### Базовый код

#### Класс Component
Является базовым классом для всех компонентов интерфейса.
Класс является дженериком и принимает в переменной `T` тип данных, которые могут быть переданы в метод `render` для отображения.

Конструктор:  
`constructor(container: HTMLElement)` - принимает ссылку на DOM элемент за отображение, которого он отвечает.

Поля класса:  
`container: HTMLElement` - поле для хранения корневого DOM элемента компонента.

Методы класса:  
`render(data?: Partial<T>): HTMLElement` - Главный метод класса. Он принимает данные, которые необходимо отобразить в интерфейсе, записывает эти данные в поля класса и возвращает ссылку на DOM-элемент. Предполагается, что в классах, которые будут наследоваться от `Component` будут реализованы сеттеры для полей с данными, которые будут вызываться в момент вызова `render` и записывать данные в необходимые DOM элементы.  
`setImage(element: HTMLImageElement, src: string, alt?: string): void` - утилитарный метод для модификации DOM-элементов `<img>`


#### Класс Api
Содержит в себе базовую логику отправки запросов.

Конструктор:  
`constructor(baseUrl: string, options: RequestInit = {})` - В конструктор передается базовый адрес сервера и опциональный объект с заголовками запросов.

Поля класса:  
`baseUrl: string` - базовый адрес сервера  
`options: RequestInit` - объект с заголовками, которые будут использованы для запросов.

Методы:  
`get(uri: string): Promise<object>` - выполняет GET запрос на переданный в параметрах ендпоинт и возвращает промис с объектом, которым ответил сервер  
`post(uri: string, data: object, method: ApiPostMethods = 'POST'): Promise<object>` - принимает объект с данными, которые будут переданы в JSON в теле запроса, и отправляет эти данные на ендпоинт переданный как параметр при вызове метода. По умолчанию выполняется `POST` запрос, но метод запроса может быть переопределен заданием третьего параметра при вызове.  
`handleResponse(response: Response): Promise<object>` - защищенный метод проверяющий ответ сервера на корректность и возвращающий объект с данными полученный от сервера или отклоненный промис, в случае некорректных данных.

#### Класс EventEmitter
Брокер событий реализует паттерн "Наблюдатель", позволяющий отправлять события и подписываться на события, происходящие в системе. Класс используется для связи слоя данных и представления.

Конструктор класса не принимает параметров.

Поля класса:  
`_events: Map<string | RegExp, Set<Function>>)` -  хранит коллекцию подписок на события. Ключи коллекции - названия событий или регулярное выражение, значения - коллекция функций обработчиков, которые будут вызваны при срабатывании события.

Методы класса:  
`on<T extends object>(event: EventName, callback: (data: T) => void): void` - подписка на событие, принимает название события и функцию обработчик.  
`emit<T extends object>(event: string, data?: T): void` - инициализация события. При вызове события в метод передается название события и объект с данными, который будет использован как аргумент для вызова обработчика.  
`trigger<T extends object>(event: string, context?: Partial<T>): (data: T) => void` - возвращает функцию, при вызове которой инициализируется требуемое в параметрах событие с передачей в него данных из второго параметра.

## Данные

Для работы приложения используются данные о товарах, выбранных позициях в корзине и покупателе, который оформляет заказ. Эти сущности описываются отдельными типами и интерфейсами.

```ts
type TPayment = 'online' | 'cash';

interface IProduct {
  id: string;
  description: string;
  image: string;
  title: string;
  category: string;
  price: number | null;
}

interface IBuyer {
  payment: TPayment;
  email: string;
  phone: string;
  address: string;
}

type ErrorsBuyer = Partial<Record<keyof IBuyer, string>>;
```

`TPayment` - тип, описывающий допустимый способ оплаты при оформлении заказа.

`IProduct` - интерфейс товара. Используется для хранения карточек каталога, отображения подробной информации о товаре и работы корзины.  
`id: string` - уникальный идентификатор товара.  
`description: string` - описание товара.  
`image: string` - путь или URL изображения товара.  
`title: string` - название товара.  
`category: string` - категория товара.  
`price: number | null` - цена товара. Значение `null` обозначает товар, недоступный для покупки.

`IBuyer` - интерфейс покупателя. Описывает данные, которые пользователь вводит при оформлении заказа.  
`payment: TPayment` - выбранный способ оплаты.  
`email: string` - email покупателя.  
`phone: string` - телефон покупателя.  
`address: string` - адрес доставки.

`ErrorsBuyer` - тип объекта с ошибками валидации формы покупателя. Строится на основе ключей `IBuyer`, поэтому автоматически расширяется вместе с моделью данных покупателя. Значением каждого свойства является текст ошибки. Если поле заполнено корректно, соответствующее свойство может отсутствовать.

## Модели данных

Модели данных отвечают за хранение состояния приложения и работу с ним. Они инкапсулируют данные и предоставляют методы доступа, изменения и проверки данных. Модели являются самостоятельными и не зависят от классов представления, коммуникации или брокера событий.

### Класс ProductCatalogModel

Назначение: хранит каталог товаров и товар, выбранный для подробного просмотра.

Зона ответственности класса:
- хранение полного списка товаров, загруженных с сервера;
- получение товаров из модели;
- поиск товара по идентификатору;
- хранение товара, выбранного для предпросмотра.

Конструктор:  
`constructor(items: IProduct[] = [], preview: IProduct | null = null)` - принимает опциональный массив товаров каталога и опциональный товар для подробного отображения.

Поля класса:  
`items: IProduct[]` - массив всех товаров каталога.  
`preview: IProduct | null` - товар, выбранный для подробного отображения.

Методы класса:  
`setItems(items: IProduct[]): void` - принимает массив товаров и сохраняет его в модели.  
`getItems(): IProduct[]` - возвращает массив товаров из модели.  
`getItem(id: string): IProduct | undefined` - принимает `id` товара и возвращает найденный товар или `undefined`, если товар не найден.  
`setPreview(item: IProduct): void` - принимает товар и сохраняет его для подробного отображения.  
`getPreview(): IProduct | null` - возвращает товар, выбранный для подробного отображения.

### Класс BasketModel

Назначение: хранит товары, которые пользователь добавил в корзину, и предоставляет методы для работы с составом заказа.

Зона ответственности класса:
- хранение выбранных товаров;
- добавление и удаление товаров из корзины;
- очистка корзины;
- подсчёт общей стоимости заказа;
- подсчёт количества товаров в корзине;
- проверка наличия товара в корзине.

Конструктор:  
`constructor(items: IProduct[] = [])` - принимает опциональный массив товаров, с которыми должна быть инициализирована корзина.

Поля класса:  
`items: IProduct[]` - массив товаров, добавленных пользователем в корзину.

Методы класса:  
`getItems(): IProduct[]` - возвращает массив товаров, находящихся в корзине.  
`addItem(item: IProduct): void` - принимает товар и добавляет его в корзину.  
`removeItem(item: IProduct): void` - принимает товар и удаляет его из корзины.  
`clear(): void` - очищает корзину.  
`getTotal(): number` - возвращает общую стоимость всех товаров в корзине.  
`getCount(): number` - возвращает количество товаров в корзине.  
`hasItem(id: string): boolean` - принимает `id` товара и возвращает `true`, если товар уже есть в корзине, иначе `false`.

### Класс BuyerModel

Назначение: хранит данные покупателя и выполняет их валидацию перед отправкой заказа.

Зона ответственности класса:
- хранение способа оплаты, адреса, телефона и email покупателя;
- частичное обновление данных без потери уже сохранённых значений;
- получение всех данных покупателя единым объектом;
- очистка данных после завершения заказа;
- проверка валидности полей формы и формирование объекта ошибок.

Конструктор:  
`constructor(data: Partial<IBuyer> = {})` - принимает опциональный объект с начальными данными покупателя и дополняет его значениями по умолчанию для всех обязательных полей.

Поля класса:  
`data: IBuyer` - объект с текущими данными покупателя. Все поля присутствуют всегда и инициализируются начальными значениями.  

Методы класса:  
`setData(data: Partial<IBuyer>): void` - принимает объект с одним или несколькими полями покупателя и обновляет только переданные значения, не затирая остальные сохранённые данные.  
`getData(): IBuyer` - возвращает текущие данные покупателя единым объектом.  
`clear(): void` - очищает данные покупателя и сбрасывает ошибки валидации.  
`validate(): ErrorsBuyer` - проверяет все поля покупателя и возвращает наружу объект с ошибками. Если поле валидно, его свойство отсутствует в результирующем объекте. Метод использует `validateField()` для последовательной проверки каждого поля и не сохраняет ошибки в состоянии модели.  
`validateField(field: keyof IBuyer): string | undefined` - принимает имя поля покупателя и возвращает текст ошибки для этого поля или `undefined`, если поле заполнено корректно.

## Слой коммуникации

Слой коммуникации отвечает за взаимодействие приложения с серверным API. Он не хранит состояние интерфейса и не управляет отображением, а инкапсулирует запросы на получение и отправку данных. Для выполнения HTTP-запросов используется композиция: коммуникационный класс принимает в конструктор объект, реализующий интерфейс `IApi`, и вызывает его методы `get` и `post`.

В слое коммуникации используются следующие типы данных:

```ts
interface IProductsResponse {
  total: number;
  items: IProduct[];
}

interface IOrderRequest extends IBuyer {
  items: IProduct['id'][];
  total: number;
}

interface IOrderResponse {
  id: string;
  total: number;
}
```

`IProductsResponse` - тип ответа сервера при запросе каталога товаров.  
`total: number` - общее количество товаров, доступных на сервере.  
`items: IProduct[]` - массив товаров.

`IOrderRequest` - тип данных, отправляемых на сервер при оформлении заказа. Основан на уже существующем интерфейсе `IBuyer` и дополняется данными о покупке.  
`items: IProduct['id'][]` - массив идентификаторов товаров, добавленных в заказ.  
`total: number` - итоговая стоимость заказа.

`IOrderResponse` - тип ответа сервера после успешного оформления заказа.  
`id: string` - идентификатор оформленного заказа.  
`total: number` - сумма подтвержденной покупки.

### Класс WebLarekApi

Назначение: выполняет запросы к API сервиса «Веб-ларёк» для получения списка товаров и отправки данных заказа.

Зона ответственности класса:
- получение каталога товаров с сервера;
- отправка данных заказа на сервер;
- типизированная работа с ответами API на уровне приложения;
- использование базового класса `Api` через интерфейс `IApi`.

Конструктор:  
`constructor(api: IApi)` - принимает объект, реализующий интерфейс `IApi`, через который будут выполняться HTTP-запросы.

Поля класса:  
`api: IApi` - объект коммуникационного слоя нижнего уровня, предоставляющий методы `get` и `post`.

Методы класса:  
`getProducts(): Promise<IProductsResponse>` - выполняет `GET` запрос на эндпоинт `/product/` и возвращает объект с общим количеством товаров и массивом товаров.  
`createOrder(order: IOrderRequest): Promise<IOrderResponse>` - принимает данные заказа, выполняет `POST` запрос на эндпоинт `/order/` и возвращает объект, подтверждающий оформление заказа на определённую сумму.
